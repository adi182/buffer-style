Buffer Style Guide

The Buffer Style Guide

The Buffer Style Guide defines a set of base components. The Buffer Style Guide is designed with these goals in mind:

  • Provide a set of core components that are currently used within the product and to allow new components to be built from those.
  • Introduce a level of thinking when building and designing.
  • Make software development easier.

Engineering Buffer

How we think of Buffer

We think of Buffer as a collection of base objects that when put together, create the Buffer product. When we explore new ideas, we default to what base objects are already available to us and if required, reuse and modify them to our needs. In some cases, these ideas lead to brand new objects that become base objects and are then added to the Buffer Style Guide.

To help frame this thinking, we think of design at Buffer as object-oriented design and utilise a variation of the BEM methodology, replacing "blocks" for "objects" and categorising "elements" as objects too. This could be thought of as the OM methodology.

Objects

Objects are standalone entities that are meaningful on their own. Examples of objects could be card, navbar, and button. These can be used together has part of a larger object or individually.

Modifiers

Modifiers are flags on objects. They modify the appearance or behaviour of objects. For example, --hidden would be a modifier that would set an object to be hidden. Modifiers are always prefixed with a double-dash: --.

Designing Buffer

Sketch and File Names

Our primary tool of choice for design in Sketch. When working on a new idea or modifying an existing idea, we begin with the base file for that particular task. For example, updating how posts are displayed within the Content tab could be achieved by duplicating the buffer-web-v1.0 Sketch document and updating its file name. Depending on the level of the change we're looking to make determines what we will increment the version number by. For example, a slight copy change might only increment the file version by .X.X, or perhaps a new page in the document might increment by .X. For any larger work or complete overhauls, we would increase the version number by a whole number completely.

To help determine who last worked on the file in case we might need some advice, we affix our initials to the end of the file name, such as buffer-web-v1.0-sd.

Layers, Groups and Symbols

Sketch is such an amazing product with fantastic layer and symbol functionality. We're super passionate about keeping the cohesion between design and engineering that we strive to keep the names of our layers and symbols the same as we would our code. In all cases, we strive to construct objects with how they might be constructed in code by using groups and symbols. For example, when using a button object in our Sketch files, we might name that button's group .button .button--primary. This way, any engineers that open our Sketch files have complete knowledge of what that object is and what they need to look for/modify when coding.

We exclusively use Open Sans as the base typeface for the Buffer product.

NavBars

NavBars are an object solely designed to hold navigational objects and typically stick to the top of the viewport.

NavBar

This is the base object for the navbar object.

Buffer

  • Library
  • Analytics
  • Help
  • Account

Profile Pictures

Profile Pictures are objects designed to display a user's profile picture.

Profile Picture

This is the base object for the profile-picture object.

Example: 
<img class="profile-picture" src="#">

Profile Picture with Icon

This is the base object for the profile-picture-with-icon object that includes a profile picture and a social network icon. It includes two separate objects (profile-picture and icon) that each inherit from their own base objects.

Example: 
<div class="profile-picture-with-icon">
  <img class="profile-picture" src="#">
  <i class="icon icon-twitter profile-picture-icon">
</div>
